CEO-UFPE · Controle de Estoque · 3 dias (15–17/06/2026) · documento para usar durante a reunião
Meta única destes 3 dias: um fluxo funcionando de ponta a ponta para mostrar ao cliente —
login → solicitante cria pedido com vários itens → almoxarife atende item por item → estoque do CEO sobe sozinho → dashboard mostra.
Profundidade > largura. O que não serve a esse fluxo, a gente corta.

1. A decisão mais importante (por que estamos mexendo na base)

O backend acabou sendo construído duas vezes, em duas bases que não se encaixam:

BaseO que éRoda o fluxo?
feat/fundacao-backendDrizzle ORM, serviços de pedido/estoque/auth, testes, docker prontoSIM — login, pedido, expedição FEFO, estoque CEO automático, tudo testado
develop / Lote-Entity / db-*SQL puro + repositórios à mãoNÃO — LoteService vazio, repositórios apontam p/ tabelas que não existem
main (hoje)versão antiga em arquivo JSONNÃO — ninguém constrói em cima dela
Decisão (ADR-0006): feat/fundacao-backend vira a base oficial, indo para main por um Pull Request revisado.
Regra nova: ninguém commita direto na main — só por PR. Vale para todo mundo, inclusive o Luiz. Isso conserta a higiene do git, não quebra.

2. Como o trabalho flui (sem pisar no pé um do outro)

  1. Ter, manhã: Luiz abre o PR da base → main (a gente revisa e aprova juntos).
  2. Ter, ~2h, TODOS juntos: subir o projeto local (1 comando, ver abaixo) e percorrer o fluxo feliz juntos. É aqui que todo mundo entende a base.
  3. Depois: cada um pega um pacote (seção 4), trabalha no seu branch, e abre PRs pequenos para a main.
  4. Luiz faz o frontend em paralelo, contra o contrato de API.

3. Subir o projeto local — 1 comando

Precisa: Docker Desktop ligado + Node 20+. A partir de projeto/backend:

# Windows (PowerShell)
.\start.ps1

# Mac/Linux/Git Bash
bash start.sh

O script sobe o Postgres, instala tudo, cria o banco, popula dados de exemplo e sobe a API em localhost:3000. Para recomeçar do zero: .\start.ps1 -Reset.

4.1 O trabalho do Wallyson na base nova

O modelo de dados v2 que estamos usando é desenho do Wallyson. Ele transformou o modelo conceitual nas 7 tabelas normalizadas (setor, usuário, produto, lote, pedido, item_do_pedido, movimentação), os enums e o índice FEFO — entregue nas stories CEO-121 (“criar script SQL de todas as tabelas conforme o diagrama E-R”) e CEO-180 (“Criar Postgres”), ambas Done.
  • A base feat/fundacao-backend reexpressou esse mesmo modelo em Drizzle (o ORM que escolhemos na ADR-0004) e construiu a aplicação por cima. Mesmas tabelas, mesmos enums, mesma ideia de FEFO — o esqueleto é o dele.
  • Os índices de performance e o trigger updated_at que ele escreveu eram melhores que a base nova (que não os tinha). Por isso foram portados para a base v2 em drizzle/0003_indices_e_trigger_wallyson.sql — vão acelerar justamente as consultas do dashboard (Pacote 2).
  • Há uma task no sprint para o Wallyson revisar/fechar esse port (a migration já está pronta como ponto de partida).
Resumindo: o trabalho dele não foi descartado — é a fundação do modelo, e a parte mais eficiente (índices/trigger) foi mantida.
