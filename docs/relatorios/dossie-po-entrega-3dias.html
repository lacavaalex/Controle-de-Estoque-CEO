2. A decisão da base (e como defendê-la)

Existem duas fundações de backend, em branches diferentes, que não se fundem (ORM diferente, nomes de entidade diferentes, camada de repositório diferente):

CritérioFundação A — develop/Lote-Entity/db-*Fundação B — feat/fundacao-backend
BancoSQL puro + repositórios Pg*Repo à mãoDrizzle ORM + schema.ts
EntidadesSolicitacao / Item / Usuario / EstoqueCeoPedido / Produto / Setor (alinhado ao modelo v2)
Roda o fluxo feliz?NÃOSIM
Testes2 arquivos (só o item legado)8 arquivos, ~960 linhas, na lógica que importa
Infra p/ rodarsem docker compose; SQL v2 sem código vivocompose + migrations + seed + JWT prontos
A prova decisiva: a Fundação A está em “split-brain” — o SQL foi atualizado para o modelo v2, mas a camada TypeScript ficou na v1. Concretamente: LoteService.ts tem 0 bytes (vazio), não há PedidoService, e os repositórios consultam tabelas que não existem. Só o caminho legado /items roda. A Fundação B implementa o modelo v2 de ponta a ponta: PedidoService.expedir() faz a alocação FEFO, a baixa no HO, a dupla movimentação saída@HO + entrada@CEO, cria/atualiza o lote-CEO (RN19) e recalcula o status derivado — tudo numa transação, com teste.
Decisão (ADR-0006): Fundação B vira a base. Vai para main por PR revisado. Fundação A é abandonada para a entrega (aproveitamos só o gatilho updated_at e índices secundários, portados por cima). Regra nova: main só por PR — ninguém commita direto, nem você.
Por que isto responde ao “não commita na main” do time: normalmente eles têm razão. O detalhe é que a main de hoje é a versão antiga abandonada — está “morta”. Esta decisão conserta a higiene: a troca entra como PR que vocês revisam, e daí em diante a regra “nada direto na main” passa a valer de verdade. Você não está quebrando o processo; está instituindo-o. Leve o ADR-0006 impresso.

Por que não as alternativas

  • Resetar develop para a B: exige force-push com 6 pessoas no meio do trabalho — risco de perder código. Descartado pelo prazo.
  • Fundir o SQL da A na B: reintroduz o split-brain v1 por ~2 arquivos aproveitáveis. Melhor portar cirurgicamente.
  • Terminar a Fundação A: seria reescrever PedidoService, FEFO, auto-alimentação e auth — exatamente o que a B já tem testado. Inviável em 3 dias.

3. Como a base funciona (arquitetura) — para você explicar

A base tem camadas limpas, o que permite a fan-out sem conflito:

  • domain/* — funções puras de regra de negócio (sem banco). Ex.: planejarExpedicaoItem (FEFO), statusDerivadoDoPedido (RN10), statusProduto (RN03–06), estadoValidadeLote (vencido/vencendo/atenção/ok). São testáveis isoladamente e devem ser reusadas — ninguém reescreve regra.
  • services/* — orquestram I/O (banco) em cima do domínio, dentro de transações. Ex.: PedidoService, EstoqueService, LoteService, AuthService.
  • repositories/Pg*Repo — acesso ao Postgres via Drizzle.
  • controller/* + routes/routes.ts — HTTP, com middleware de autenticar (JWT) e exigir (RBAC).
  • auth/* — jwt, bcrypt (senha.ts), guards RBAC (podeVerSetor, podeProcessarPedidos, podeEditarEstoque, podeCriarPedido).
Conceito que confunde (e você vai precisar explicar): no pedido, “origem” = quem pede (ex.: CEO) e “destino” = o almoxarifado HO. Mas os bens fluem ao contrário: saem do HO e entram no CEO. Por isso a expedição lê os lotes do HO e gera saída@HO + entrada@CEO. Está documentado no próprio código (direcaoExpedicao).

5. Os 5 pacotes (detalhe técnico para alocar)

Cada pacote é fatia vertical (domínio → serviço → rota → teste), baixo conflito, entra por PR. Sem dono fixo — decidam na reunião. Abaixo, o suficiente para você explicar cada um e estimar dificuldade.

P1 Estoque do CEO: consumo + ajuste  (EP03-03 / EP03-04 · CEO-238, CEO-239)

  • Objetivo: o gestor do CEO abate consumo clínico e corrige saldo por recontagem.
  • Construir: registrarConsumo(...) e ajustarInventario(...) — transacional, gera Movimentacao tipo consumo/ajuste (delta). Consumo > saldo → erro “Faça ajuste de inventário…”. Rotas POST /lotes/:id/consumo, PATCH /lotes/:id/ajuste.
  • Onde apoiar: LoteService.ajustarQuantidade já existe — mesmo padrão de transação, só novos tipos de movimentação. RBAC: reusar podeEditarEstoque.
  • Dificuldade: média-baixa. Bom para quem precisa de escopo fechado (ex.: Vinícius).

P2 Dashboards / KPIs  (EP05-01/04/06 · CEO-248, CEO-251, CEO-253)

  • Objetivo: endpoints de agregação que o front consome por perfil.
  • Construir: GET /dashboard?setorId= com total de produtos, lotes vencendo (≤30/≤60d), críticos, pedidos pendentes e demanda represada (top itens aguardando_reposicao). GET /setores/:id/pedidos?status= para a fila e contadores.
  • Onde apoiar: reusar statusProduto e estadoValidadeLote de domain/estoque.ts — não recalcular na mão.
  • Dificuldade: média (é leitura/agregação, sem escrita → baixo risco de conflito). Encaixa em quem fez dashboard/cadastro (ex.: Bruno).

P3 Ciclo de vida do lote: segregação  (EP07-01/02 · CEO-213, CEO-214)

  • Objetivo: expor estados de validade + ação de segregar lote vencido.
  • Construir: LoteService.segregar(loteId, responsavelId, observacao) → estado segregado + data_segregacao + Movimentacao tipo segregacao. POST /lotes/:id/segregar + GET /setores/:id/lotes-segregados. Confirmar por teste que segregado some da expedição.
  • Onde apoiar: estadoValidadeLote já existe; LoteService já tem entrada/ajuste como molde.
  • Dificuldade: média-baixa, escopo fechado. Bom para quem já mexeu com Lote (ex.: José Romildo).

P4 Pedidos: robustez + teste de integração  (EP04-06/08 · CEO-245, CEO-247)

  • Objetivo: fechar pontas e provar o caminho feliz ponta-a-ponta.
  • Construir: o que mais falta — teste de integração HTTP: criar pedido → expedir item → GET /setores/:ceo/estoque mostra o saldo subindo. Abas/contadores por status e visibilidade por perfil. Casos de borda (expedir item já processado, expedir linha livre deve barrar, item indisponível vira aguardando_reposicao).
  • Onde apoiar: PedidoService.test.ts e PgPedidoRepo.test.ts já existem — estender.
  • Dificuldade: média-alta (exige entender o fluxo todo). Para o dev mais forte de backend (ex.: Kaique).

P5 Catálogo + seed + e-mail  (EP02 / EP04-07 · CEO-246)

  • Objetivo: polir o catálogo, garantir uma demo coesa e a ponte de e-mail.
  • Construir: validações faltantes no CRUD de produto (categoria/unidade inválida, bloquear remoção com lote ativo = RN13) com teste; caprichar o seed de demonstração (um pedido pronto p/ expedir na frente do cliente, lotes vencendo, item “Não Tem”); e-mail ao criar pedido (cortável — pode ser um stub que loga).
  • Onde apoiar: ProdutoService, src/db/seed.ts.
  • Dificuldade: média; o e-mail é o primeiro a cair. Encaixa em quem domina services/catálogo (ex.: Alex, se ficar no backend).

6. Como rodar e testar local

Precisa: Docker Desktop ligado, Node 20+. A partir de projeto/backend, um comando:

# Windows (PowerShell)
.\start.ps1
# Mac / Linux / Git Bash
bash start.sh
# recomeçar o banco do zero:  .\start.ps1 -Reset   (ou  bash start.sh --reset)

O script sobe o Postgres (porta 5433, de propósito — a 5432 já é usada por outro projeto), instala deps, aplica migrations, popula o seed e sobe a API em localhost:3000.

  2. O achado: backend feito 2 vezes; uma roda, a outra não. Mostrar a tabela da seção 2.

Se vier objeção…

  • “Não pode commitar na main.” → Concordo. Por isso é um PR revisado, e a partir daí a regra “só PR” vale pra todos. A main de hoje é a versão antiga abandonada; estamos consertando, não quebrando.

  • “Não conheço Drizzle.” → Por isso o kickoff de 2h juntos. A regra de negócio está em funções puras (domain/*), fáceis de ler; você quase não toca no ORM.
  • “E o estoque do CEO, alguém fez?” → A parte difícil (alimentar o CEO automaticamente na expedição) já está pronta e testada. Falta só consumo/ajuste (P1).
