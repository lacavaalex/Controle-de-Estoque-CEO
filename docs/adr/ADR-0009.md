Número do ADR: 0009
Título: Frontend SPA em React + Vite, com Context API e React Router (sem framework full-stack)
Data: 25/06/2026
Responsável: Alex Lacava Jr — implementação dominante no histórico (`projeto/frontend/src`); co: Luiz Taiguara (PO), Bruno Silva
Status: Aceito (consolidação retroativa de decisão já implementada)

#Contexto
O sistema tem um backend próprio em Express (API REST) e precisa de uma interface web rica: múltiplos perfis com rotas distintas (sidebar varia por perfil/setor), tabelas com filtros, formulários de pedido multi-item, tela de triagem do agente e dashboards. Era preciso escolher a tecnologia de frontend e o quanto de "framework" adotar — desde React puro até um framework full-stack como Next.js.

#Decisão
Adotamos uma **Single Page Application (SPA) em React 19 + Vite 8**, consumindo a API do backend via `fetch` (cliente em `src/api/`). Para roteamento, **React Router 7**; para estado de sessão/autorização, a **Context API** nativa (`src/auth/AuthContext.jsx`, `RequireAuth.jsx`) — sem biblioteca de estado externa (Redux/Zustand). Testes com **Vitest + Testing Library** (jsdom). O build do Vite é servido como estático (Nginx) em produção (ADR-0010).

#Justificativa
- **Separação limpa front/back.** Backend já é uma API REST independente (ADR-0007); uma SPA que a consome mantém as responsabilidades separadas e permite que front e back evoluam e sejam testados isoladamente.
- **Vite = DX rápida.** Dev server com HMR quase instantâneo e build otimizado, sem a configuração pesada de bundlers antigos. Acelera o ciclo de iteração de UI, relevante no prazo curto.
- **Context API é suficiente.** O estado verdadeiramente global é pequeno (usuário autenticado + token + perfil/setor). Para isso, Context resolve sem o boilerplate e a curva de Redux. Estado de tela fica local nos componentes. "Use a plataforma antes de adicionar dependência."
- **Sem Next.js de propósito.** Não há necessidade de SSR/SEO (é uma ferramenta interna autenticada) nem de rotas de API no front (o backend Express já existe). Next adicionaria um runtime Node e um modelo mental (server components, etc.) sem ganho para este caso.
- **Testabilidade.** Testing Library + Vitest cobrem componentes e fluxos (53 testes), rodando em jsdom sem navegador — entram no CI (ADR de Qualidade) em segundos.

#Alternativas Consideradas
- **Next.js (full-stack React).** SSR, file-based routing e rotas de API. Descartado: SSR/SEO são irrelevantes para um app interno autenticado, e as rotas de API duplicariam o backend Express. Peso e complexidade sem retorno.
- **Redux / Zustand para estado.** Padrão para estado global complexo. Descartado: o estado global aqui é mínimo; Context cobre. Introduzir agora seria otimização prematura.
- **Angular / Vue.** Frameworks completos e válidos, mas a equipe tinha mais familiaridade com React, e o ecossistema React + Vite atende o escopo. Decisão de produtividade do time.
- **CRA (Create React App).** Praticamente descontinuado e mais lento que Vite. Descartado.

#Consequências
- **Sem SSR:** primeira carga baixa o bundle JS; aceitável para ferramenta interna em desktop/tablet (RNF de responsividade mira ≥768px). Se algum dia precisar de SEO/landing pública, reavaliar.
- **Context tem limites:** se o estado global crescer muito (muitos contexts aninhados, re-render amplo), pode ser preciso memoização cuidadosa ou migrar partes para uma store. Hoje não é gargalo; fica registrado como ponto de atenção.
- **Estado de servidor é buscado sob demanda** via `fetch` (sem React Query): simples, mas sem cache automático/revalidação. Para o volume atual de telas funciona; uma camada de data-fetching seria a evolução natural se a app crescer.
- **Identidade visual:** a UI deve seguir a marca (cor primária — ver nota no ADR de UI/identidade; o oficial CIn é `#DB1E2F`). Componentização reutilizável já em uso.
