Número do ADR: 0011
Título: IA como ferramenta de engenharia e gestão — Agent Skills, hooks e MCP do Jira
Data: 25/06/2026
Responsável: Luiz Taiguara (PO), com a equipe
Status: Aceito (consolidação retroativa de prática adotada ao longo do projeto)

#Contexto
O projeto foi conduzido por uma equipe de 7 pessoas em 45 dias, com um PO acumulando descoberta de domínio, escrita de backlog, integração de branches e parte da implementação. Decidiu-se, de forma deliberada, usar **assistentes de IA (Claude Code) como ferramenta de engenharia e de gestão** — não para "gerar código sozinho", mas para acelerar tarefas estruturadas e disciplinar processos (descoberta de requisitos, quebra em issues, diagnóstico de bugs, revisão de UI, gestão do board). Esta ADR registra **quais ferramentas de IA entraram no fluxo e por quê**, porque essa escolha de método afeta como o trabalho foi produzido e é parte legítima do processo de desenvolvimento.

#Decisão
Adotamos três classes de ferramentas de IA, versionadas/configuradas no repositório:

1. **Agent Skills do ecossistema `mattpocock/skills`** (em `.agents/skills/`), cada uma um prompt-disciplina reutilizável:
   - **`grill-me`** — entrevista o autor sobre um plano/decisão até esgotar a árvore de decisões. Usada para stress-testar planos (incluindo o desenho desta entrega e do agente).
   - **`to-prd`** — transforma o contexto de uma conversa num PRD publicável no tracker. Apoiou a estruturação do backlog v2 a partir do fluxo real do almoxarifado.
   - **`to-issues`** — quebra um PRD/plano em issues independentes e "pegáveis". Apoiou a geração das User Stories e épicos no Jira (CEO-216…CEO-272).
   - **`diagnose`** — loop disciplinado de diagnóstico (reproduzir → minimizar → hipótese → instrumentar → corrigir → teste de regressão). Usado no diagnóstico do bloqueio de IMAP (rastro: `METAPROMPT-DIAGNOSTICO.md`, `docs/relatorios/diagnostico-imap.py`) que fundamentou o ADR-0004.
   - **`caveman`** — modo de comunicação comprimido (~-75% tokens) para iterações rápidas.

2. **`Impeccable`** (https://github.com/pbakaus/impeccable) — skill/**hook** de revisão de frontend que roda a cada edição de arquivo de UI, registrando `findings` por arquivo (`.impeccable/hook.cache.json` presente em 6 níveis do repo). Funciona como um revisor de design contínuo durante o desenvolvimento do front.

3. **Revisão de código assistida (`/code-review`)** — revisão automatizada do *diff* antes do commit/PR, caçando bugs de correção e oportunidades de simplificação/reuso. Complementa (não substitui) o code review humano do Pull Request: a IA faz uma primeira passada sistemática, e o revisor humano decide. Usada inclusive na própria preparação desta entrega (CI + ADRs).

4. **MCP do Jira** (Model Context Protocol) — servidor que conecta o assistente ao Jira, permitindo ler e manter o board (criar/transicionar issues, sincronizar status com o código, auditar pendências) a partir do contexto de desenvolvimento, sem alternar de ferramenta manualmente.

#Justificativa
- **Disciplina, não atalho.** As skills codificam *processos* (entrevistar requisitos, quebrar em issues, diagnosticar com método) que um time sênior seguiria de qualquer forma — a IA os torna repetíveis e rápidos. O valor está no rigor imposto, não em "pular etapas".
- **Rastreabilidade do backlog.** `to-prd` + `to-issues` ajudaram a derivar um backlog estruturado (37 US, épicos, Gherkin) a partir da descoberta real, mantendo a ligação requisito→issue.
- **Qualidade de UI contínua.** O hook do Impeccable trouxe feedback de design no momento da edição, complementando o code review humano no frontend.
- **Dupla rede de revisão.** `/code-review` (lógica/correção, no diff) e Impeccable (design, na UI) dão uma primeira varredura sistemática antes do PR; o code review humano segue como autoridade final. Camadas que se somam, não se substituem.
- **Gestão do board sem fricção.** O MCP do Jira permitiu manter status × código sincronizados e auditar o board (ver relatórios de status em `docs/relatorios/`), reduzindo o descompasso entre o que está feito e o que o board mostra.
- **Transparência acadêmica.** Registrar o uso de IA como ferramenta é honesto e alinhado à engenharia de software moderna, que trata tooling (incluindo IA) como parte do processo.

#Alternativas Consideradas
- **Não usar IA / usar de forma ad-hoc (sem skills versionadas).** Perde-se reprodutibilidade e disciplina; cada interação reinventa o processo. Descartado em favor de skills versionadas no repo.
- **Gestão do Jira 100% manual.** Funciona, mas o board ficou grande (legado v1 + v2) e o custo de manter status × código sincronizados à mão é alto; o MCP reduziu esse atrito.
- **Revisão de UI só no code review humano.** Mantida como prática, mas o hook do Impeccable adianta achados no momento da edição, antes do PR.
- **Não registrar o uso de IA em ADR.** Descartado: omitir uma decisão de método relevante enfraqueceria a documentação do processo (justamente o que esta entrega valoriza).

#Consequências
- **Dependência de tooling externo:** skills e MCP dependem de configuração local (chaves, tokens). O token do Jira já exigiu manutenção (renovação) — um custo operacional conhecido. As skills, por serem versionadas, mitigam o risco de "funciona só na máquina de quem instalou".
- **Supervisão humana é obrigatória.** IA propõe; a equipe revisa, decide e assume. Todo código entra por Pull Request com code review humano (e agora CI — ver ADR de Qualidade). Nenhuma decisão arquitetural foi delegada à IA: as ADRs registram autoria humana real (inferida do histórico Git).
- **Curva e ruído.** Skills mal usadas geram saída irrelevante; o time aprendeu quando cada uma se aplica. O `caveman` e o `grill-me`, por exemplo, servem a momentos distintos (compressão vs. aprofundamento).
- **Reprodutibilidade parcial:** as skills estão no repo (`.agents/skills/`, `.claude/skills/`), mas reproduzir exige o mesmo runtime de agente — registrado como limitação.
