Número do ADR: 0006
Título: Drizzle ORM como camada de acesso a dados (sobre o driver `pg`)
Data: 25/06/2026
Responsável: Luiz Taiguara (PO) — implementação dominante no histórico (`src/db`, `drizzle/`); co: Alex Lacava, Kaique Bonfim
Status: Aceito (consolidação retroativa de decisão já implementada)

#Contexto
Definida a tecnologia de banco (PostgreSQL, ADR-0001) e o boilerplate Express + TypeScript, faltava decidir COMO o código TypeScript conversa com o Postgres: SQL cru via driver `pg`, um query builder, ou um ORM. O sistema tem modelagem relacional não-trivial (Produto × Lote 1:N, Pedido cabeçalho + N itens, Movimentação polimórfica) e regras que exigem **transações** (expedição: baixa em lote + saída no HO + entrada no CEO precisam ser atômicas). A equipe trabalha em TypeScript e valoriza segurança de tipos do schema até a query.

#Decisão
Adotamos o **Drizzle ORM** (`drizzle-orm` + `drizzle-kit`) sobre o driver `pg`, com o schema declarado em TypeScript (`src/db/schema.ts`) como fonte única de verdade, migrations versionadas geradas por `drizzle-kit generate` (`drizzle/0000…0006`) e aplicadas por `drizzle-kit migrate`. O cliente é um `pg.Pool` único compartilhado (`src/db/client.ts`), e `db.transaction(...)` é usado nos pontos críticos de consistência.

#Justificativa
- **Type-safety de ponta a ponta.** O schema em TS gera os tipos das entidades (`entities/index.js` deriva de `schema.ts`); uma coluna renomeada quebra o `tsc` em vez de explodir em runtime. Reduz uma classe inteira de bugs de persistência.
- **Migrations versionadas e revisáveis.** `drizzle-kit` gera SQL legível, commitado no repo (`drizzle/*.sql` + `meta/_journal.json`). Cada mudança de schema entra por PR e é auditável — essencial para um time de 7 pessoas com integração via Pull Request.
- **Transações de primeira classe.** `db.transaction` expõe um executor (`Tx`) que os repositórios compõem (ex.: promoção rascunho→pedido, expedição) — materializa o ACID que o ADR-0001 só prometia. Ver `LoteService` (transação na movimentação de estoque) e `PgPedidoRepo`.
- **Próximo do SQL, sem "magia".** Diferente de ORMs pesados, o Drizzle é uma fina camada tipada sobre SQL; o time lê a query que será executada. Alinhado ao princípio de **baixo acoplamento e previsibilidade** discutido em engsoftmoderna.info (cap. Projeto de Software).

#Alternativas Consideradas
- **SQL cru via `pg`.** Máximo controle, zero tipagem do resultado e migrations manuais. Descartado: o custo de manter tipos e migrations à mão num schema que evoluiu para 7 migrations seria alto e propenso a erro.
- **Prisma.** ORM popular e maduro, mas com runtime/engine próprio mais pesado, geração de cliente em etapa separada e abstração mais distante do SQL. Descartado: mais peso e "magia" do que o projeto precisa; Drizzle entrega type-safety com modelo mental de SQL.
- **TypeORM / Sequelize.** Padrão Active Record/Data Mapper com decorators; histórico de incompatibilidades com ESM/TS estrito e tipagem mais frouxa. Descartado por ergonomia inferior no nosso stack ESM + TS.

#Consequências
- **Acoplamento ao Drizzle no repositório.** As implementações `Pg*Repo` dependem da API do Drizzle. Mitigado pela arquitetura de portas/adaptadores (ADR-0007): os *services* dependem das interfaces `I*Repo`, não do Drizzle — trocar o ORM afetaria só a camada de adaptadores.
- **Disciplina de migrations.** Mudou o `schema.ts` → tem que gerar a migration e commitá-la. A regeneração já foi exercitada num merge sensível (renumeração da migration do EP08 0004→0005, regenerada via `drizzle-kit` em vez de editar snapshot à mão).
- **Curva de aprendizado pequena** para quem sabe SQL; a equipe absorveu rápido.
- **Ganho de graça:** o seed (`src/db/seed.ts`) e o CI usam o mesmo `db:migrate`, garantindo que o banco de teste seja idêntico em estrutura ao de produção.
